# Mobile UI Refresh Design for YTTSCRB

## Goal

Исправить мобильную вёрстку главного экрана YTTSCRB и одновременно заметно улучшить визуальное качество интерфейса без изменения продуктовой логики, API-контрактов или структуры данных.

## Current Problems

1. Форма ввода ссылки остаётся в одну строку на узких экранах, из-за чего кнопка выталкивает layout вправо.
2. Контейнеры и внутренние отступы недостаточно адаптивны для мобильных экранов.
3. Hero-блок визуально слишком простой и не даёт ощущение polished SaaS-интерфейса.
4. Карточка результата рабочая, но слабая по иерархии: summary, transcript и CTA не имеют достаточно чётких визуальных уровней.
5. Длинный текст транскрипта и длинные URL потенциально могут ухудшать читаемость на мобильных устройствах.

## Chosen Approach

Выбран сбалансированный вариант:

- починить mobile layout;
- усилить hero и form card;
- улучшить визуал статусов и result card;
- не превращать экран в полноценный маркетинговый landing;
- не менять backend или пользовательский сценарий.

Это даёт заметный UX/UI-apgrade при низком риске и минимальном затрагивании архитектуры приложения.

## Scope

В рамках этой правки меняются только фронтенд-файлы текущего экрана:

- `resources/js/App.vue`
- при необходимости `resources/css/app.css`
- при необходимости базовая оболочка `resources/views/welcome.blade.php`

Вне scope:

- изменения API;
- новые backend-фичи;
- изменение workflow;
- изменение статусов задач;
- новые страницы;
- история, авторизация, экспорт в новые форматы.

## UI Design

### 1. Layout and Responsiveness

- Главный контейнер должен иметь более безопасные мобильные `px` и адаптивные `py`.
- Форма на маленьких экранах переходит в вертикальный stack.
- Input и submit button занимают полную ширину на mobile.
- На `sm` и выше форма может возвращаться к горизонтальной компоновке.
- Все основные карточки должны ограничивать ширину и не провоцировать горизонтальный скролл.

### 2. Hero Upgrade

- Добавить более выразительную тёмную сцену с мягкими glow/gradient-акцентами.
- Усилить типографику заголовка и подзаголовка.
- Добавить компактные value/trust pills, например про fast summary, transcript output, no signup.
- Hero должен оставаться компактным и product-first, а не marketing-heavy.

### 3. Form Card

- Форма размещается в glass-like карточке с мягкой границей, blur/transparent overlay и лучшей глубиной.
- Поле ввода получает более комфортную высоту и состояние focus.
- CTA-кнопка выглядит более премиально: градиент/сильнее hover/тень, но без избыточной анимации.
- Состояние загрузки остаётся понятным и доступным.

### 4. Status and Result Card

- Badge состояний должны быть более различимыми визуально.
- `pending` и `processing` состояния должны быть чище и понятнее на первом взгляде.
- Summary должен быть оформлен как отдельный выделенный блок.
- Transcript должен быть вынесен в отдельную скроллящуюся панель с хорошей читаемостью.
- Download и Retry CTA должны быть заметнее и удобнее на мобильных устройствах.

### 5. Mobile Readability Safeguards

- Использовать безопасные переносы и `break-words` там, где это нужно.
- Не допускать сжатия кнопок до неудобных размеров.
- Текстовые блоки должны иметь комфортную line-height и размеры шрифта.
- Transcript container должен сохранять вертикальный scroll внутри себя, а не ломать страницу.

## Data Flow and Behavior

Поведение приложения остаётся прежним:

1. Пользователь вставляет YouTube URL.
2. Отправляется `POST /api/transcribe`.
3. UI поллит `GET /api/transcribe/{id}`.
4. По статусу рендерятся pending / processing / completed / failed состояния.
5. При `completed` пользователь видит summary, transcript и download CTA.
6. При `failed` пользователь может повторить запрос.

Никаких изменений в бизнес-логике или сетевых вызовах не планируется.

## Error Handling

- Ошибки остаются в текущем UX-потоке, но error card может получить более аккуратное оформление.
- Ошибки не должны ломать layout на мобильных устройствах.
- Длинный текст ошибок должен переноситься внутри контейнера.

## Testing Strategy

Так как изменение фронтенд-поведения затрагивает разметку, реализация должна идти через TDD-подход настолько, насколько это практично для текущего проекта:

1. Сначала проверить существующую фронтенд-структуру и доступные test setup.
2. Добавить/обновить тест(ы) на ключевые UI-инварианты, если в проекте уже есть подходящий стек для Vue component testing.
3. Если такого test setup ещё нет и его добавление disproportionate для данной правки, явно зафиксировать ограничение и провести верификацию через сборку проекта и ручную проверку responsive layout.

Минимальные инварианты для проверки:

- форма имеет mobile-first stacking;
- CTA не выталкивает layout;
- completed state продолжает показывать summary/transcript/download;
- failed state продолжает показывать retry.

## Implementation Notes

- Предпочтительно решить задачу в `App.vue` через Tailwind utility classes, без тяжёлой кастомной CSS-архитектуры.
- `app.css` использовать только для ограниченного количества глобальных декоративных эффектов, если они улучшают читаемость шаблона.
- Сохранять текущую Vue-логику polling и состояния, меняя только presentation layer.

## Risks and Mitigations

### Risk: visual overdesign

Митигация: сохранить один экран, один основной CTA и без маркетинговых секций ниже fold.

### Risk: mobile fix introduces desktop regression

Митигация: использовать mobile-first классы Tailwind и проверять layout на mobile + desktop breakpoints.

### Risk: decorative styles reduce readability

Митигация: контентные карточки держать достаточно тёмными и контрастными, декоративный glow оставлять фоновым.

## Acceptance Criteria

1. На мобильной ширине форма больше не вызывает горизонтальный скролл.
2. Input и button корректно stack-ятся на узких экранах.
3. Основной экран визуально выглядит заметно более polished.
4. Summary и transcript читаются лучше, чем в текущей версии.
5. Статусы `pending`, `processing`, `completed`, `failed` по-прежнему корректно отображаются.
6. API-интеграция и polling не изменяются.
